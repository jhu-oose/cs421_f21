Homework 4

caution
  • You are expected to work on this homework as a team in your (project) group.
  • Due: Wednesday September 29nd at 11pm EST (Baltimore time).
  • This assignment is worth 50 points.

Learning Objectives​

Objectives
  • Relational database management systems, SQL, DAO, ORM
  • SQLite, JavaSpark, ORMLite, JUnit, OKHttp
  • Basic web API development using SparkJava

Task​

Get the starter code for homework 4 from JBApp repo. Clone the repo since this is the first time you are copying the files. Inside hw4_starter folder, you will find a Gradle Java project with the following structure:

JBApp/hw4_starter/src
└── /main/java
|   └── 
|       ├── model
|       │   └── Employer.java
|       |   └── Job.java
|       └── Main.java
├── /test/java
    ├── EmployerTest.java
    └── JobTest.java

  1. Model classes Employer.java & Job.java: These two classes model Employer and Job entities in our software. The class Employer has a number of fields and methods and annotated using ORMLite annotations. Job.Java also has a number of fields and methods in it, including a employerId employer field which will be used to relate a specific instance of a job to a specific employer. Take time to study the two files and get yourself familiarized with them!

  2. Main.java: If you run Main.java, it will start a local SparkJava server on your machine on http://localhost:7000. It also defines an API endpoint /employers. If you start the server and visit http://localhost:7000/employers on your browser, it will show a list of employers to you in JSON format (assuming you have employer records saved in the employers table within JBApp database). Take time to study Main.java and develop an understanding of it.

  3. EmployerTest.java: This is a JUnit test class. There are two inner classes (i.e. nested classes) inside EmployerTest:

    • EmployerORMLiteDaoTest: there are already five unit tests written for you in this class to give you an idea how to write test cases in JUnit, and how to make use of dao object to do various CRUD operations on a table (employers table in this case.) These test methods run and assert different CRUD operations on the employers table.

      TODO #1

      Read the comments and add more test cases inside EmployerORMLiteDaoTest for the listed scenarios. Note for each listed scenario you need to write a test case in a separate test method and make use of JUnit assert methods appropriately. If you want to do something and you are not sure how to do it and cannot find your answer in the given test cases, consult JUnit 5 user guide.

    • EmployerAPITest: This is the second inner class inside EmployerTest class that will contain our test cases that test the /employers API endpoint of our JBApp. For now, there is only one test method here named testHTTPGetEmployersEndpoint() that is empty.

      TODO #2

      Read the comments and complete the implementation of testHTTPGetEmployersEndpoint(). At minimum, you need to verify the return status code of the http get request you send using OkHttp. Ideally though, you want to make sure the received response in JSON format indeed is correct and contains the expected content, but it is not a requirement for this homework!

  4. Now, go back to Job.java and annotate the Job class using ORMLite annotations similar to the way Employer class is annotated. But you need to annotate it such that you will then be able to create jobs table using TableUtils, the very same way we create Employer table in the getEmployerORMLiteDao() method of Main.java

    tip

    In particular, you need to make sure the foreign key constraint gets created on the employer field that references a particular employer in the employers table. You might want to check out the reading assignment on Connecting Job and Employer classes and/or consult ORMLite documentation and its various annotations!

    TODO #3

    Complete annotating the Job class using ORM annotations!

  5. Head back to Main.java to add a /jobs endpoint (similar to what we have done in employers endpoint). When a http get request is sent to this API endpoint, it must return the list of all jobs stored in the jobs table in JSON format and set the status code to 200.

    TODO #4

    Complete implementation of /jobs API endpoint in Main.java

  6. Finally, move over to JobTest.java. It is structured similar to EmployerTest.java. JobTest.java is a JUnit test class with two inner classes (i.e. nested classes):

    • EmployerORMLiteDaoTest: You need to write at least 12 different test cases (i.e., test methods) that would test C(reate)/U(pdate)/D(elete) operations on the jobs tables using a ORMLite DAO. You need to write at least 4 test cases for each of the CUD operations. Agian, what we are trying to do here is very similar to what is already done in EmployerTest and specifically EmployerORMLiteDaoTest. So, if you are not sure how/where to start, go back and study EmployerORMLiteDaoTest to get some inspirations!

    • JobAPITest: This is the second inner class inside JobTest class that will contain our test cases that test the /jobs API endpoint of our JBApp. For now. there is only one test method here named testHTTPGetJobsEndpoint() that is empty.

      TODO #6

      Read the comments and finish the implementation of testHTTPGetJobsEndpoint(). At minimum, you need to verify the return status code of the http get request you send using OkHttp. Ideally though, you want to make sure the received response in JSON format indeed is correct and contains the expected content, but it is not a requirement for this homework!

tip

To complete this homework, you will need to work with different libraries and tools. If you intend to do something and are not sure how to proceed, refer to the website/documentation/user guide/manual of that tool/library. Remember our note on Self-Efficacy & Self-Directed Learning. Researching/finding things on your own and getting into the habit of reading&learning about different tools/technologies/libraries on your own are invaluable assets found in the arsenal of any (decent) software engineer!
